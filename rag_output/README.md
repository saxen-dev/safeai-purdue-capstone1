# RAG Output

This directory contains the retrieval-augmented generation (RAG) artifacts produced by the pipeline's chunking and retrieval stages. These are the benchmark results and deployable package for Brain 1 (the knowledge layer).

## Directory structure

```
rag_output/
  retrieval_test_results.json   # 12-query benchmark (P@3, P@5, MRR per query)
  build_report.json             # Build metadata, timing, corpus stats, aggregate metrics
  child_chunks.json             # All 1,695 child chunks with metadata and contextual headers
  bm25_corpus.json              # BM25 sparse index corpus (tokenized)
  embeddings_pubmedbert.npy     # PubMedBERT 768-dim embeddings (gitignored, regenerated)
  faiss_index_flat.bin          # FAISS flat inner-product index (gitignored, regenerated)
  chroma_db/                    # ChromaDB development index (gitignored, regenerated)
  brain1_package/               # Mobile-ready export package (24.11 MB total)
    package_manifest.json       # Package metadata and checksums
    parent_chunks.json          # 1,478 parent chunks with full clinical metadata
    child_chunks.json           # 1,695 child chunks sized for embedding search
    bm25_inverted_index.json    # BM25 index for on-device sparse search
    faiss_index.bin             # FAISS index for on-device dense search (gitignored)
```

## What is tracked in git

| File | Tracked | Reason |
|------|---------|--------|
| `retrieval_test_results.json` | Yes | Benchmark proof — reviewers need this |
| `build_report.json` | Yes | Corpus stats, timing, aggregate metrics |
| `child_chunks.json` | Yes | The actual chunks used for retrieval |
| `bm25_corpus.json` | Yes | BM25 index (JSON, human-reviewable) |
| `brain1_package/*.json` | Yes | Deployable mobile package |
| `*.npy`, `*.bin` | No | Binary indices, regenerated by `pipeline/retriever.py` |
| `chroma_db/` | No | Development-only vector DB, regenerated |

## Key files explained

### retrieval_test_results.json

The 12-query clinical benchmark. Each query entry contains:

- `query_id`, `query`, `query_type` (dosage_lookup, contraindication, semantic, etc.)
- `precision_at_3`, `precision_at_5`, `mrr` per query
- Per-result details: rank, chunk ID, parent ID, chunk type, preservation level, keyword coverage, relevance flag, content preview, source pages

**Aggregate results** (from `build_report.json`):

| Metric | Value |
|--------|-------|
| Mean Precision@3 | 0.944 |
| Mean Precision@5 | 0.967 |
| Mean MRR | 0.944 |
| Perfect P@3 (= 1.0) | 11 / 12 queries |

### build_report.json

Pipeline build metadata including:

- **Corpus:** 1,695 child chunks, 412,592 total tokens, avg 243 tokens/child
- **Type breakdown:** 1,471 narrative, 60 dosing table, 142 other table, 21 evidence, 1 clinical
- **Preservation breakdown:** 1,196 standard, 438 high, 61 verbatim
- **Timing:** Embedding took ~551s on CPU (PubMedBERT-768)
- **Models:** Primary embedding: PubMedBERT-SBERT (768-dim), comparison: BGE-M3 (1024-dim)
- **Retrieval config:** top-20 per modality, RRF k=60, cross-encoder reranking on top-10

### brain1_package/

The mobile-ready export. Total size: 24.11 MB (FAISS 4.97 MB + child chunks 5.50 MB + parents 10.80 MB + BM25 2.85 MB). This is the artifact that ships to the Android app for offline clinical Q&A.

## Regenerating binary artifacts

If the gitignored files are missing (FAISS index, embeddings, ChromaDB), they are automatically rebuilt when the pipeline runs:

```bash
python run_pipeline.py --preset who-malaria --pdf /path/to/malaria.pdf
```

The JSON files in this directory are the ground truth. The binary indices are derived from them.

## Cross-referencing with source PDFs

To verify retrieval results against the original guidelines:

1. Open `retrieval_test_results.json` and pick a query (e.g., Q04: artesunate-amodiaquine dosing)
2. Check `source_pages` in each result to find the PDF page numbers
3. Open the corresponding PDF and navigate to those pages
4. Compare the `content_preview` against the actual PDF text
